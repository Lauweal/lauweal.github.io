##### 前言

&emsp;&emsp;这个篇纯属个人的开发体验，算不上真实的业务开发中的架构设计。我自始至终都算是一个react的框架使用者,个人还是比较感谢这个框架的，毕竟给了我饭碗。恰饭的同时，它给我最大的帮助就是从15到17的大部分版本我都接触了,让我接触到了从公司层面接触不到的一个技术向产品的演进过程。毕竟目前很多时候都是以先满足业务为主，我们讨论的是直面用户的业务架构，比如用一个个小的产品做出业务架构的闭环。但是个人比较难的去感受我用这个包或者库,它解决了什么东西（狭义上说解决的就是一个需求）。当然宏观的东西总是让人产生厉害的感觉,错误的也会以为自己也很厉害。这里还是主要说说react,vue作为参考毕竟没写多少,随意说原理也不合适。所以，想想看看就好该搬的砖还是接着搬，恰饭继续。

##### 技术演进的体现点

###### 出发点即体现点

&emsp;&emsp;此处会提到vue友军勿打,大家都是恰js饭的人都是自己人哈。想要得到的东西最终都会以现在成果中,有的架构变动是从开发者出发的,有的架构是从解决技术问题出发的。这两个不是直指vue和react两个框架,更多的是两个框架变动中每个小变动的考虑。下面提点大家都知道的案例吧！

+ vue的jsx语法变更以及微前端
&emsp;&emsp;vue2.x版本更新中曾经出现了jsx语法的支持,这个算是一个站在开发者角度做的架构变动了。由于它站的生态就是浏览器端,所以用jsx的语法更像是写html,同时能从react开发者中获取一部分人员。类似的案例启示也能体现在微前端上,微前端的出现也是顺应着开发者出现的,在早起的前端生态中js与jq混搭写是不存在问题的,jq在设计实现上只是一个js的domapi的封装,是没有抽象的数据流业务模型的,所以任何一个开发者都可以很好的接入进来只要js底子比较好（没有理论模型的壁垒就是普世的知识。后期大家发现其实前端没有一个很好的业务实现的模型,就开始在js之上构建了三大框架实现比较稳定的数据流方案。绕回到微前端上,它有一个点就是为了融合三大框架的差异让一个公司中允许并存三大框架,实现方式不同拖累了公司太多的精力做兼容技术方案了,将项目拆除能也能适应敏捷开发（项目管理是个头痛的事）,只能从框架和技术的角度去适应人了。不过这些方案统一的不是业务模型,属于整合方案。

+ vue的proxy实现迁移
&emsp;&emsp;说到proxy的迁移它属于一个技术优化的架构演进,核心点在于对象内存的优化。js的对象属性变更这一点,背后的动作可能很少体现在我们的开发中,常常也会像下面这样写。

``` typescript
    const schema = { count: 1 };
    schema["status"] = false;
    const instance = new Schema({ count: 1 });
    instance["status"] = false;
```
